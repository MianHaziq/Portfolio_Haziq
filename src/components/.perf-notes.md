# Scroll/animation performance contract

Read before adding animations. These rules keep the site at 60fps.

1. **No static `will-change`.** Never hard-code `will-change` in inline styles
   on repeated/discrete elements (cards, tech tiles, dots, skill bars). It pins
   each element to a permanent GPU layer; dozens of them = layer explosion and
   the "heavy/hanging" feel. GSAP (`force3D`) and framer-motion already promote
   layers transiently during a tween, which is what you want. `will-change` is
   only justified on the few elements that animate *continuously*: ScrollProgress
   bar, hero orbs (CSS float), the experience timeline line (scrub), hero content
   parallax.

2. **`backdrop-filter` is expensive.** Use it sparingly. It is intentionally
   OFF for `.tech-card` (~36 on screen) — they use a flat translucent bg instead.
   Don't add it to grid items that repeat.

3. **One scroll loop.** Lenis is driven by GSAP's ticker in `SmoothScroll.tsx`
   and pipes into `ScrollTrigger.update`. Don't add a second rAF/scroll smoother
   or native `scroll-behavior: smooth` (they fight and stutter).

4. **Don't sprinkle `ScrollTrigger.refresh()`.** It's a synchronous global
   reflow. Triggers self-position on creation; the single post-Lenis refresh +
   `document.fonts.ready` refresh in `SmoothScroll.tsx` covers font-swap shifts.

5. **One transform owner per element.** Never animate `transform` from both a
   CSS keyframe and GSAP on the same node (the hero orbs used to — they fought).
